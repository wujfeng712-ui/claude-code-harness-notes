# 00 · 核心心智模型：Agent = Model + Harness

> 读完本篇，你应该能用一句话向别人解释「一个 coding agent 是怎么工作的」。

## 一个公式

整个 `learn-claude-code` 项目，以及几乎所有 coding agent，都建立在同一个公式上：

```
┌─────────────────────────────────────────────────────────────┐
│                                                               │
│        Agent   =   Model        +        Harness              │
│                  （大脑/会推理）      （工具·权限·上下文·记忆·协调）   │
│                   ↑ 不可控          ↑ 这才是工程师能掌控的部分        │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

- **Model** 是大脑：它会推理、会决定「下一步该调用哪个工具」。但你改不了它的权重，只能给它喂上下文。
- **Harness** 是大脑周围的一切工程：工具怎么注册、权限怎么把关、上下文满了怎么办、跨会话怎么记住、多个 agent 怎么协作。**这才是你能动手的地方，也是 agent 强弱的真正分水岭。**

## 一个永不改变的循环

Model 和 Harness 之间，靠一个极简的循环连接：

![核心循环：模型决策 → 执行工具 → 结果回填](../images/diagrams/01-core-loop.svg)

```python
while stop_reason == "tool_use":
    response = call_model(messages)        # ① 模型决策
    if response.stop_reason != "tool_use":
        break                              # 模型说「我说完了」→ 退出
    for tool_call in response.tool_calls:  # ② 执行工具
        result = TOOL_HANDLERS[tool_call.name](**tool_call.input)
        messages.append(tool_result(result))  # ③ 结果回填
```

**这是全项目唯一、也是最重要的设计主张**：这段循环从第 1 课到第 20 课**一字未改**。20 个机制全部是在它**周围**叠加，而不是改写它本身。

## 为什么这件事重要

如果每加一个功能都要改循环，循环很快会变成一坨几千行、谁都不敢碰的怪物。`learn-claude-code` 的做法是：

```
                    ┌──────── 不变的内核 ────────┐
   UserPromptSubmit │   call model → tool_use?   │ Stop
   ──────钩子──────►│   ├─ 是 → 执行工具 → 回填    │◄────钩子─────
   PreToolUse       │   └─ 否 → 返回             │ PostToolUse
                    └────────────────────────────┘
        ▲                                              ▲
        └── 所有新机制都「挂」在这些扩展点上，不进内核 ──┘
```

**s04 的 hooks（钩子）是让这一切成立的工程支点**。引入钩子后，循环成为稳定内核，后续每个机制（权限、日志、压缩、记忆…）都靠「挂钩」接入，而不是往循环里塞 `if-else`。

> 记住这一句，后面 20 课就都是它的注脚：
> **「挂在循环上，不写进循环里。」**

## 三条贯穿全程的工程原则

读后面任何一课，都能看到这三条原则的影子：

1. **信任代码，不信任模型** —— 权限、路径校验都在代码层硬强制，而不是「相信模型不会乱来」。
2. **便宜的先做，贵的后做** —— 压缩分层、错误恢复分层，能用 0 次 API 解决的绝不调用 LLM。
3. **用文件系统做持久化与解耦** —— 任务、记忆、消息邮箱、worktree 全是文件，天然支持跨会话恢复和多 agent 协作。
