# s02 · Tool dispatch — from 1 tool to N tools

> **Motto: add a tool by adding a handler; the loop stays put.**

## What this lesson solves

The s01 loop only ran a single hard-coded tool. To support many tools without touching the loop each time you add one, s02 turns the tool-execution line from hard-coded into table dispatch: a new tool = one schema + one handler, and the loop itself never changes.

## How the mechanism works

The only change is on the tool-execution line: from hard-coded `run_bash(...)` to table dispatch.

```python
result = TOOL_HANDLERS[block.name](**block.input)
```

- `TOOL_HANDLERS` is a dispatch map from tool name to handler function.
- Adding a tool just means registering it in the table; the loop logic stays the same.
- `safe_path()` checks that file-type tools cannot escape the workspace, preventing directory traversal.

## Key insights

- Pulling the "which tool to run" decision out of the loop into a table decouples the loop from the number of tools.
- File-type tools must validate paths — the more powerful a tool, the more the code layer has to backstop it.
