# s04 · Hooks — the turning point

> **Motto: hang it on the loop, don't write it into the loop.**

## What this lesson solves

s03 hard-coded the permission check into the loop. If every new mechanism (permissions, logging, compaction, memory…) keeps stuffing `if-else` into the loop, the loop quickly becomes a monster nobody dares touch. s04 introduces hooks: move the checks out of the loop so the loop becomes a stable core, and any extension plugs in by "hanging on." **This lesson is the engineering pivot of the whole project.**

## How the mechanism works

Move the `check_permission()` that s03 hard-coded into the loop out into a `permission_hook`, and change the loop to:

```python
blocked = trigger_hooks("PreToolUse", block)
```

`HOOKS` is an event registry; four events let any extension plug in by hanging on:

- `UserPromptSubmit` — after user input, before the model call (inject context)
- `PreToolUse` — before a tool runs (permissions, logging)
- `PostToolUse` — after a tool runs (output checks, side effects)
- `Stop` — before the loop exits (cleanup, can force a continuation)

## Key insights

- With hooks in place, the loop turns from "logic that keeps swelling" into "stable core + extension points."
- This is the prerequisite that lets every later mechanism (from s05 on) layer cleanly — they all hang on these four events.
