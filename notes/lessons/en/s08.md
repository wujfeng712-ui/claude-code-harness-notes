# s08 · Context Compaction

> **Motto: Run the cheap passes first, the expensive ones last.**

## What this lesson solves

After continuous work, `messages` balloons until it bursts the context. This lesson makes room with a layered compaction pipeline whose core idea is **run the cheap passes first, the expensive ones last** — never call the LLM for something that 0 API calls can solve.

## How the mechanism works

A four-layer pipeline, ordered from lowest to highest cost:

```
before each LLM call:
  L3 tool_result_budget  spill big results to disk (>30KB)   ─┐
  L1 snip_compact        trim the middle (keep head 3 + tail 47) ├─ 0 API calls
  L2 micro_compact       swap old results for placeholders     ─┘
        │
        ▼  still > threshold (50K tokens)?
  L4 compact_history      LLM full summary                     ── 1 API call
        │
        ▼  API returns 413?
  reactive_compact        aggressive trim, retry once          ── emergency
```

Order matters: **L3 must come before L2** — otherwise big results get replaced by placeholders before they're spilled to disk. `compact_history` saves the full conversation into `.transcripts/` so details can be recovered later, and the summary must preserve 5 kinds of information: current goal, important findings, files changed, remaining work, user constraints.

## Key insights

- Layering is essentially cost ordering: run all the 0-API trims first, only pay 1 API call for a summary when those don't suffice.
- L3 before L2 is a hard constraint: spill big results to disk first, then replace with placeholders, or you lose data.
- A summary isn't arbitrary compression: it must preserve the 5 kinds — goal / findings / changes / remaining / constraints.
