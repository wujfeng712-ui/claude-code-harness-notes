# s09 · Memory

> **Motto: Compaction loses details — you need one layer that doesn't.**

## What this lesson solves

s08's compaction inevitably loses detail ("tab vs spaces" gets flattened to "code-style preference"), and it can't remember across sessions. The memory system adds a **lossless** persistence layer beyond compaction — it remembers across compactions and across sessions.

## How the mechanism works

The memory system has three stages:

![Memory's three stages: Selection → Extraction → Consolidation](../../../images/diagrams/04-memory.svg)

- **Selection**: at the start of each turn, an LLM side-query picks ≤5 relevant memories from the store and injects them into the current turn.
- **Extraction**: at the end of each turn, new memories are extracted from the **pre-compaction snapshot** (not from compacted data, to keep information intact).
- **Consolidation**: when memory files accumulate to ≥10, an LLM pass is triggered to dedupe, merge, and retire stale items.

Storage is file-based: the `.memory/MEMORY.md` index stays resident in SYSTEM (~100 tokens/turn), and each memory is a `.md` with frontmatter. Four kinds: `user` (who is who) / `feedback` (how to do things) / `project` (what happened) / `reference` (where to find things).

## Key insights

- Extraction must run on the **pre-compaction snapshot**, or you record the already-lossy compacted version.
- Index resident, body on demand: MEMORY.md stays resident cheaply; individual memories are injected only per Selection results.
- Memory self-cleans: hitting the threshold triggers Consolidation to dedupe and retire, so it doesn't accumulate into a mess.
