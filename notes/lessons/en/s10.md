# s10 · Runtime System-Prompt Assembly

> **Motto: The prompt is assembled, not hard-coded.**

## What this lesson solves

Writing the system prompt as one hard-coded string keeps it from adapting to the real environment — whether a memory store exists, whether a skill is available, none of it can be reflected. This lesson upgrades the SYSTEM prompt into a **config assembled from real state**.

## How the mechanism works

The SYSTEM prompt is upgraded from a hard-coded string into a **config assembled from real state**. `PROMPT_SECTIONS` are concatenated on demand (e.g. the memory section loads only if `.memory/MEMORY.md` exists), using `json.dumps` (not `hash()`) as the cache key to avoid redundant in-process re-assembly. Whether a section loads is decided by **real file/tool state**, not by guessing keywords in the messages.

```
real state ── file exists? tool available? ──▶ assemble_system_prompt
PROMPT_SECTIONS ─ concatenated on demand ─────▶  assembled result
                                                  │ json.dumps as cache key
                                                  ▼  reuse on cache hit, no re-assembly
```

## Key insights

- Which section loads is decided by **real file/tool state**, not by guessing keywords in the messages.
- The cache key uses `json.dumps` rather than `hash()`, so the same state hits stably and isn't re-assembled.
