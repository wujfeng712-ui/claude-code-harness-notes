# s11 · Error Recovery — Three Layers, Each Its Own Job

> **Motto: An error is a retry's starting point, not the end.**

## What this lesson solves

In distributed systems, truncation, limit overruns, and transient failures are the norm, not the exception. Cramming all recovery logic into one place gets fragile fast. This lesson splits recovery into three layers — each handling only its own class of failure, so adding a new recovery path never disturbs the others. `RecoveryState` also prevents the same failure from being "recovered" over and over.

## How it works

Three layers, each with its own job — from bottom to top they handle "transient faults → single-request exceptions → why did we stop":

- **Tool layer `with_retry`**: transient errors (429/529) get exponential backoff + jitter; after 3 consecutive 529s, switch to a fallback model.
- **API layer try/except**: classifies single-request exceptions. `max_tokens` → bump the token limit (8K→64K) and inject a continuation hint; `prompt_too_long` → call `reactive_compact` for one aggressive compaction.
- **Loop layer `stop_reason`**: in the main loop, decide "why did we stop" and whether to continue.

```
Loop layer    stop_reason check (end_turn / tool_use / max_tokens…)
  ▲
API layer     try/except classify: max_tokens→bump token+continue; prompt_too_long→compact
  ▲
Tool layer    with_retry: 429/529 backoff+jitter; 3 consecutive 529s→fallback model
```

`RecoveryState` records which recovery actions were already attempted, so the same failure isn't repeatedly patched.

## Key insights

- The value of layering is "isolating change": each layer recognizes only its own error class, so new recovery paths stay independent.
- Truncation (`max_tokens`) is not a failure but an "unfinished sentence" — the right move is to bump tokens and continue, not redo.
- Switching to a fallback model is a last resort, not a first choice: back off and retry first; only escalate after repeated failures.
