# s12 · Task Graph (DAG) — A File-Backed Dependency Graph

> **Motto: Split into tasks, order them, persist them.**

## What this lesson solves

A big goal can't be swallowed in one bite — it must be split into small tasks, ordered for execution, and recoverable across sessions. This lesson models tasks as a directed acyclic graph (DAG): one JSON file per task, declaring upstream dependencies via `blockedBy`. No database needed — file persistence gives cross-session recovery for free, and an `owner` field prevents duplicate claims in multi-agent scenarios.

## How it works

Each task is a JSON record (`id / subject / status / owner / blockedBy`) stored as `.tasks/{id}.json`. `can_start()` checks whether all upstreams in `blockedBy` are completed; only then may a task be claimed. On completion, it scans and reports the downstream tasks it just unblocked.

## Key insights

- The file *is* the database: one JSON per task gives cross-session recovery for free, with no real database required.
- The `owner` field is the cornerstone of multi-agent collaboration — it stops two agents from claiming the same task.
- `blockedBy` + `can_start()` make execution order *declarative*: the scheduler just checks whether dependencies are satisfied.
