# s13 · Background Tasks — Slow Ops Don't Block

> **Motto: Slow ops go to the background, the agent moves on.**

## What this lesson solves

Slow ops like install / build / test make the agent sit and wait, with the main loop stuck on one long command. This lesson pushes slow ops into a background daemon thread: the agent gets an immediate placeholder result and moves on to other work; once the background finishes, the real result is injected asynchronously.

## How it works

Slow ops (commands containing install/build/test keywords, or when the model explicitly passes `run_in_background`) are pushed into a daemon thread, and the main loop keeps running. When the background finishes, the result is injected as a `<task_notification>` — **crucially, it does not reuse the original tool_use_id**.

```
tool call ──┬─ returns placeholder result immediately ("started in background…") → agent loops on
            │
            └─ daemon thread runs the slow op …… done
                        │
                        ▼
              <task_notification> injected as a standalone event next turn
```

Why not reuse the original `tool_use_id`: the original tool call already returned a placeholder result, so the pairing is closed. The background completion is a **standalone event** and must be injected as a new message — otherwise it breaks the tool_use/tool_result pairing.

## Key insights

- "Placeholder result + async injection" gives the single-threaded agent loop concurrency — while the loop itself stays unchanged.
- Background completion is a standalone event; you can't stuff it back into a closed tool pairing — this is s13's easiest trap.
- The trigger can be heuristic (keywords) or explicit (`run_in_background`); both take the same background path.
