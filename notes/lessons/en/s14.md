# s14 · Cron Scheduler — Produce Work on a Schedule

> **Motto: Produce work on a schedule; decouple schedule from run.**

## What this lesson solves

Some work needs to "run automatically when the time comes," not be hand-triggered each time. This lesson adds cron scheduling, fully decoupling "when it should run" from "how it actually runs" — the scheduler thread only produces work on a schedule, the agent loop only consumes it, and neither blocks the other.

## How it works

Four layers of decoupling keep scheduling and execution fully separate:

```
scheduler thread (poll every sec, match cron) → cron_queue (buffer triggered) → queue processor (deliver when agent idle) → loop consumes & injects
```

- **Scheduler thread**: polls every second, matching the cron expression against the current time.
- **cron_queue**: buffers triggered tasks, decoupling production from consumption.
- **queue processor**: delivers only when the agent is idle, avoiding interruption of work in progress.
- **Loop consumes & injects**: the main loop injects the triggered task as a new round of input.

`minute_marker = "%Y-%m-%d %H:%M"` prevents re-triggering within the same minute; durable task definitions are written to `.scheduled_tasks.json` for cross-restart recovery, while session-only tasks stay in memory.

## Key insights

- The payoff of decoupling schedule from run: the scheduler thread never calls the model directly — it just posts to the queue, leaving the execution pace entirely to the agent's idle windows.
- `minute_marker` is the key to dedup — polling every second means the same minute matches many times, so a marker is required to prevent re-triggering.
- The line between durable and session-only is simply "do we write `.scheduled_tasks.json` or not."
