# s16 · Team Protocols — A Handshake With a State Machine

> **Motto: Teammates need a protocol.**

## The problem this lesson solves

In s15 teammates talk to each other in loose free text, so requests and replies don't line up — who answered, which request they answered, whether it's even the right kind of answer, all left to luck. s16 upgrades loose text into a **structured protocol**: correlate requests and replies by ID, type-check to prevent crossed wires, and replace a hard exit with a graceful-shutdown handshake.

## How the mechanism works

- `pending_requests[request_id]` correlates a request with its reply by ID, fixing the "mismatched answer" problem.
- `match_response()` does type validation: a `shutdown_response` won't be mistaken for approving a `plan_approval_request`.
- After receiving a `shutdown_request`, a teammate leaves its idle loop, enters dispatch, and replies with a `shutdown_response` — a **graceful shutdown** instead of a hard kill.

```
Lead ──request(id=42, type=plan_approval)──▶ Teammate
pending_requests[42] = plan_approval               │ handle
Lead ◀──response(id=42, type=plan_approval)──────── reply
match_response(42, plan_approval) ✓ id matches, type matches → approve

Lead ──request(type=shutdown)──▶ Teammate(idle → dispatch)
Lead ◀──response(type=shutdown)── graceful exit, not a hard kill
```

## Key insight

- ID + type double-checking turns "can chat" into "can reconcile": a reply must match its request, and the type can't be crossed.
- Shutdown is itself a request-reply handshake, giving a teammate a chance to wrap up instead of being cut off.
- The protocol is the dividing line between loose text and reliable collaboration.
