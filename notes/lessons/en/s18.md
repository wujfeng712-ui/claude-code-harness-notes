# s18 · Worktree Isolation — Each Does Its Own, No Conflicts

> **Motto: Each does its own, no conflicts.**

## The problem this lesson solves

When several teammates develop in parallel, if they all edit files in the same working directory their changes overwrite and conflict with each other. s18 uses `git worktree` to give each task its **own directory + own branch**, so parallel development truly doesn't interfere at the file level.

## How the mechanism works

- `git worktree add` creates a separate directory `.worktrees/{name}/` + branch `wt/{name}` per task, bound via the task's `worktree` field.
- A teammate's bash/read/write run automatically in that cwd, so **parallel development files don't collide**.
- `validate_worktree_name()` allows only `[A-Za-z0-9._-]` to prevent path traversal.
- Before removal it checks for uncommitted changes and refuses to delete if any exist (unless `discard_changes` is set explicitly).

```
task A (worktree=feat-a) ─► .worktrees/feat-a/  branch wt/feat-a
task B (worktree=feat-b) ─► .worktrees/feat-b/  branch wt/feat-b
                              │
   teammate A's bash/read/write land in feat-a/ ── no cross-pollution
```

## Key insight

- Binding task ↔ directory pushes "parallel collaboration" down from the logical layer to the filesystem layer, so conflicts are physically separated.
- Name validation + a pre-delete dirty check are the safety boundary of isolation: prevent traversal, prevent deleting uncommitted work by accident.
- Worktrees naturally give each teammate its own branch; merging is decided higher up, with no blocking during development.
