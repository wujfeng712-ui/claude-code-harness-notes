# s19 · MCP Plugin — External Tools via a Standard Protocol

> **Motto: External tools via a standard protocol.**

## The problem this lesson solves

Through s18, every tool was hardcoded inside `code.py`. To plug in external services (a database, search, a third-party API) you can't edit the source each time. s19 uses **MCP (Model Context Protocol)** as a standard protocol, so external tools slot into the tool pool through a uniform discovery + call interface, leaving the source untouched.

## How the mechanism works

- MCP acts as a standard protocol; external services connect through two interfaces: **discovery + call**.
- `assemble_tool_pool()` rebuilds the tool pool every round, merging built-in tools with MCP tools.
- Each MCP tool gets a prefix `mcp__{server}__{tool}` to prevent same-named tools from different servers from colliding.
- A change in the tool set invalidates the prompt cache and forces a rebuild.

```
built-in tools ┐
               ├─► assemble_tool_pool() rebuilds each round ─► handed to LLM
MCP server A ─discovery─► tool x ──► mcp__A__x ┘
MCP server B ─discovery─► tool x ──► mcp__B__x  (prefix prevents collision)
                          │
                       call interface executes
```

## Key insight

- A standard protocol downgrades "adding a tool" from "edit the source" to "connect a service" — external capabilities become plug-and-play.
- The prefix namespace `mcp__{server}__{tool}` is the key to coexisting servers, avoiding crossed wires between same-named tools.
- Rebuilding the tool pool every round has a cost: any change to the tool set invalidates the prompt cache — the trade-off between flexibility and cache hits.
