# s03 · 权限管线 — 执行前的三道闸

> **Motto：信任代码，不信任模型。**

## 这一课解决什么问题

模型能调 Bash，就可能调出危险命令。不能靠「相信模型不会乱来」，必须在代码层做强制检查。s03 在工具执行前串联三道闸：硬禁列表、规则匹配、人工确认，把安全边界从模型挪到代码。

## 机制怎么实现

工具执行前依次走三道闸：`DENY_LIST` → `PERMISSION_RULES` → `ask_user()`。

![06-permission-gates](../../images/diagrams/06-permission-gates.svg)

<details><summary>📄 ASCII 版（终端可读）</summary>

```
工具调用 block
      │
      ▼
┌───────────────┐  命中    ┌──────────────┐
│ ① DENY_LIST   │ ───────► │  直接拒绝     │
│  硬禁列表      │          └──────────────┘
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐  无规则   ┌──────────────┐
│ ② RULES 匹配  │ ───────► │   直接放行    │
└───────┬───────┘          └──────────────┘
        │ 命中规则
        ▼
┌───────────────┐  y / N   ┌──────────────┐
│ ③ ask_user()  │ ───────► │ 放行 / 拒绝   │
└───────────────┘          └──────────────┘
```

</details>

- ① `DENY_LIST`：硬禁列表，命中直接拒绝。
- ② `PERMISSION_RULES`：规则匹配，无规则则直接放行。
- ③ `ask_user()`：命中规则的交给人工确认，y 放行 / N 拒绝。

## 关键洞见

- 安全不是模型的责任，是 harness 的责任——在代码层硬强制。
- 三道闸由严到松排序：先看绝对禁止的，再看规则，最后才问人。
