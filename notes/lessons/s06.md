# s06 · 子 Agent（上下文隔离）

> **Motto：大任务拆小，每个小任务干净的上下文。**

## 这一课解决什么问题

主 Agent 一旦把所有探索、试错、中间产物都堆进同一个 `messages[]`，上下文很快被噪声污染、撑到溢出。子 Agent 把一个大任务交给一条**全新、干净**的对话去跑，只把最终结论带回来，主线对话不被中间过程拖累。

## 机制怎么实现

`task` 工具 spawn 出一个拥有**全新 `messages[]`** 的子 Agent，让它跑自己的循环，跑完**只回传最后一条摘要**，中间过程全部丢弃。如同「开一个新终端，干完关掉，结果带回主终端」。

```
主 Agent messages[] ─── task("研究 X") ──▶ 子 Agent 全新 messages[]
                                              │ 自己的循环：探索/试错/产出
                                              ▼
主 Agent ◀──────── 只回传最后摘要 ───────── 中间过程全部丢弃
```

⚠️ 关键区分：隔离的是**上下文**，不是**权限**——子 Agent 的工具调用仍要过 `PreToolUse` 钩子。子 Agent 没有 `task` 工具，防止无限递归。

## 关键洞见

- 隔离的是上下文，不是权限：子 Agent 依旧受 `PreToolUse` 钩子约束。
- 子 Agent 不持有 `task` 工具，从机制上杜绝无限递归。
- 「只回传摘要、丢弃过程」是用一次性的脏上下文换主线的干净。
