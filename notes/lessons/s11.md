# s11 · 错误恢复 — 三层各司其职

> **Motto：错误不是终点，是重试的起点。**

## 这一课解决什么问题

分布式系统里，截断、超限、临时故障是常态，而不是异常。把所有恢复逻辑揉在一处会越来越脆。这一课把错误恢复拆成三层，各层只管自己那类故障，新增一条恢复路径不会牵动其他层。`RecoveryState` 还能防止同一次失败被反复恢复。

## 机制怎么实现

三层各司其职，从底到顶分别处理「瞬态故障 → 单次请求异常 → 为什么停下」：

- **工具层 `with_retry`**：瞬态错误（429/529）做指数退避 + 抖动重试；连续 3 次 529 时切换到备用模型。
- **API 层 try/except**：分类处理单次请求异常。`max_tokens` → 升级 token 上限（8K→64K）并注入续写提示；`prompt_too_long` → 调用 `reactive_compact` 激进压缩一次。
- **循环层 `stop_reason`**：在主循环里判断「为什么停下」，决定是否继续。

```
循环层   stop_reason 判停（end_turn / tool_use / max_tokens…）
  ▲
API 层   try/except 分类：max_tokens→升 token 续写；prompt_too_long→压缩
  ▲
工具层   with_retry：429/529 退避+抖动；连续 3 次 529→切备用模型
```

`RecoveryState` 记录已尝试的恢复动作，避免对同一次失败重复施救。

## 关键洞见

- 分层的价值在于「隔离变化」：每层只认自己那类错误，新增恢复路径互不干扰。
- 截断（`max_tokens`）不是失败而是「没说完」——正确动作是升 token 续写，而非重来。
- 切备用模型是兜底而非首选：先退避重试，连续失败才升级。
