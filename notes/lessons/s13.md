# s13 · 后台任务 — 慢操作不阻塞

> **Motto：慢操作丢后台，Agent 继续处理。**

## 这一课解决什么问题

install / build / test 这类慢操作会让 Agent 干等，主循环被一个长命令卡死。这一课把慢操作丢进后台 daemon 线程，Agent 立刻拿到一个占位 result 继续处理别的事；后台完成后再把真实结果异步注入对话。

## 机制怎么实现

慢操作（命令含 install/build/test 关键词，或模型显式传 `run_in_background`）被丢进 daemon 线程，主循环继续往下跑。后台完成后，结果以 `<task_notification>` 格式注入对话——**关键是不复用原 tool_use_id**。

```
工具调用 ──┬─ 立即返回占位 result（"已在后台启动…"）→ Agent 继续循环
           │
           └─ daemon 线程跑慢操作 …… 完成
                       │
                       ▼
              <task_notification> 作为独立事件注入下一轮
```

为什么不复用原 `tool_use_id`：原 tool call 已经回了占位 result，配对已闭合；后台完成是一个**独立事件**，必须以新消息注入，否则会破坏 tool_use/tool_result 的配对。

## 关键洞见

- 「占位 result + 异步注入」让单线程的 Agent 循环获得了并发能力，而循环本身不变。
- 后台完成是独立事件，不能塞回已闭合的 tool 配对里——这是 s13 最容易踩的坑。
- 触发条件可以是启发式（关键词）也可以是显式（`run_in_background`），两者都走同一条后台路径。
