# s14 · Cron 调度 — 按时间表生产工作

> **Motto：按时间表生产工作，调度与执行解耦。**

## 这一课解决什么问题

有些工作需要「到点自动跑」，而不是用户每次手动触发。这一课引入 cron 调度，把「什么时候该跑」和「实际怎么跑」彻底解耦——调度线程只负责按时间表生产工作，Agent 循环只负责消费，互不阻塞。

## 机制怎么实现

四层解耦，让调度和执行完全分开：

```
调度线程(每秒轮询匹配cron) → cron_queue(缓冲已触发) → queue processor(Agent空闲时交付) → 循环消费注入
```

- **调度线程**：每秒轮询，用 cron 表达式匹配当前时间。
- **cron_queue**：缓冲已触发的任务，解耦生产与消费。
- **queue processor**：在 Agent 空闲时才交付，避免打断正在进行的工作。
- **循环消费注入**：主循环把触发的任务作为新一轮输入注入。

`minute_marker = "%Y-%m-%d %H:%M"` 防止同一分钟内重复触发；durable 任务定义写进 `.scheduled_tasks.json` 以便跨重启恢复，session-only 任务只留在内存。

## 关键洞见

- 调度与执行解耦的好处：调度线程从不直接调用模型，只往队列里投，执行节奏完全交给 Agent 空闲窗口。
- `minute_marker` 是去重的关键——每秒轮询意味着同一分钟会匹配多次，必须用标记防重触发。
- durable vs session-only 的分野，就是「写不写 `.scheduled_tasks.json`」。
