# s15 · Agent 团队 — 文件收件箱

> **Motto：一个搞不定，组队来。**

## 这一课解决什么问题

单个 Agent 串行干活，遇到能并行拆分的大任务就力不从心。这一课让多个 Agent 组队：Lead 派活、队友并行干、干完回报。关键是要有一条可靠的通信通道，保证每条消息只被处理一次。

## 机制怎么实现

`MessageBus` 基于文件收件箱 `.mailboxes/{agent}.jsonl`：

- **发消息** = append 一行 JSON。
- **读消息** = read + unlink（**消费式**，保证每条只处理一次）。

![多 Agent 协作：Lead 经 MessageBus 与队友通信，队友自治认领任务](../../images/diagrams/05-teams.svg)

Lead spawn 出队友线程，给队友一套精简工具集（bash / read / write / send_message）。队友并行工作，完成后发 summary 回 Lead 的 inbox；Lead 下一轮检查 inbox，把队友的回报注入自己的历史。

## 关键洞见

- 「消费式读取」（read + unlink）是只处理一次的关键——消息读走就删，天然防重复消费。
- 文件收件箱无需消息中间件：append + unlink 就是一个最小可靠队列，复用了 s12 的「文件即持久化」思路。
- 队友拿到的是精简工具集而非全集——能力边界即安全边界。
