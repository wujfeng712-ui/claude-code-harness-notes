# s16 · 团队协议 — 带状态机的握手

> **Motto：队友之间要有约定。**

## 这一课解决什么问题

s15 的队友之间靠松散文本通信，请求和回复「对不上号」——谁回的、回的是哪条、是不是答非所问，全靠运气。s16 把松散文本升级为**结构化协议**：用 ID 关联请求与回复，用类型校验防止串台，并把硬退出换成**优雅关机**的握手。

## 机制怎么实现

- `pending_requests[request_id]` 用 ID 关联请求与回复，解决「对不上号」。
- `match_response()` 做类型校验：`shutdown_response` 不会被误当成 `plan_approval_request` 批准。
- 队友收到 `shutdown_request` 后从 idle loop 进入 dispatch，回复 `shutdown_response`，实现**优雅关机**而非硬退出。

```
Lead ──request(id=42, type=plan_approval)──▶ Teammate
pending_requests[42] = plan_approval               │ 处理
Lead ◀──response(id=42, type=plan_approval)──────── 回复
match_response(42, plan_approval) ✓ 对上号、类型对 → 批准

Lead ──request(type=shutdown)──▶ Teammate(idle → dispatch)
Lead ◀──response(type=shutdown)── 优雅退出，而非硬 kill
```

## 关键洞见

- ID + 类型双重校验，把「能聊」升级为「能对账」：回复必须对得上请求，类型还不能串。
- 关机也是一次请求-回复握手，让队友有机会收尾，而不是被硬退出打断。
- 协议是松散文本与可靠协作之间的那道分界线。
