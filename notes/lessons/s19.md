# s19 · MCP 插件 — 外接工具，标准协议

> **Motto：外接工具，标准协议。**

## 这一课解决什么问题

到 s18 为止，工具都是内置在 `code.py` 里的。要接入外部服务（数据库、搜索、第三方 API），不能每次都改源码。s19 用 **MCP（Model Context Protocol）**作为标准协议，让外部工具通过统一的发现 + 调用接口插进工具池，源码不动。

## 机制怎么实现

- MCP 作为标准协议，外部服务通过 **discovery + call** 两个接口接入。
- `assemble_tool_pool()` 每轮重组工具池，把内置工具和 MCP 工具合并。
- 为每个 MCP 工具生成前缀 `mcp__{server}__{tool}`，防止不同 server 的同名工具冲突。
- 工具集变化会使 prompt 缓存失效、需要重建。

```
内置工具 ┐
         ├─► assemble_tool_pool() 每轮重组 ─► 交给 LLM
MCP server A ─discovery─► tool x ──► mcp__A__x ┘
MCP server B ─discovery─► tool x ──► mcp__B__x  （前缀防冲突）
                          │
                       call 接口执行
```

## 关键洞见

- 标准协议把「加工具」从「改源码」降级为「连服务」，外部能力即插即用。
- 前缀命名空间 `mcp__{server}__{tool}` 是多 server 共存的关键，避免同名工具串台。
- 每轮重组工具池是代价：工具集一变，prompt 缓存就失效——灵活性和缓存命中之间的取舍。
